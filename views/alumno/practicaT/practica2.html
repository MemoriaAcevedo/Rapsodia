Recomendaciones
  1. Solicitar ejemplos al corrector (Product Owner) sobre el funcionamiento del software.
  2. Tener claro las reglas de negocio que las funcionalidades deben cumplir.
  3. Utilizar nombres representativos para las pruebas y comenzar con la palabra should (debería). Ejemplo deberiaEncontrarClientePorId.
  4. Contemplar las 3 etapas para confeccionar una prueba unitaria: Preparación, Acto, Afirmación.
  5. Verificar el cumplimiento de las características de una prueba unitaria: Atómica, Independiente, Inicua, Rápida, Informativa.
  6. Asegurarse que cada prueba automatizada utilizando Test Driven Development, permita crear el mínimo código que satisfaga el comportamiento y regla de negocio de la funcionalidad.
